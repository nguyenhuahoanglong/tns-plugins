# Followup Diff

How `/publish-review followup <wi>` decides what's resolved.

## Inputs

- Prior state file `.CodeReview/.{branch}.publish.json` with `mustFixSlugs: [...]`
- New report `.CodeReview/{branch}.md` (re-generated by `/code-review-pro` after dev's fixes)

## Algorithm

```python
prior_slugs = set(state["mustFixSlugs"])
new_parse = parse_must_fix(new_report_path)  # scripts/parse_must_fix.py
new_slugs = set(item["slug"] for item in new_parse["mustFix"])

resolved = sorted(prior_slugs - new_slugs)        # gone in N+1 → fixed
remaining = sorted(prior_slugs & new_slugs)        # still present
new_items = sorted(new_slugs - prior_slugs)        # appeared in N+1 (regressions or newly surfaced)

return {
    "resolved": resolved,
    "remaining": remaining,
    "new": new_items,
    "resolved_count": len(resolved),
    "total_count": len(prior_slugs),
    "remaining_count": len(remaining)
}
```

## Banner rules (used by phase 5 of the publish pipeline)

| Case | Banner |
|---|---|
| `resolved_count == total_count` AND `len(new) == 0` | `**Iteration {n}** — All Must Fix resolved ({total}/{total}). Ready for re-review.` |
| `resolved_count > 0` AND `len(new) == 0` | `**Iteration {n}** — Resolved {resolved}/{total}: {resolved_slugs}.` |
| `resolved_count > 0` AND `len(new) > 0` | `**Iteration {n}** — Resolved {resolved}/{total}: {resolved_slugs}. New: {new_slugs}.` |
| `resolved_count == 0` AND `len(new) > 0` | `**Iteration {n}** — No items resolved. New: {new_slugs}.` |
| `resolved_count == 0` AND `len(new) == 0` | `**Iteration {n}** — No changes since last review.` (rare; suggests dev hasn't pushed new commits) |

## Bullet section after banner

- All-resolved case: replace `**Must Fix:**` block with `_(none remaining — see attachment for full report)_`
- Otherwise: render only `remaining` + `new` items as numbered bullets. Drop `resolved` items (they're acknowledged in the banner).

## Edit-in-place rationale

Discussion clarity beats history granularity. Edit-in-place keeps the WI Discussion section from accumulating duplicate review comments across N iterations of fix-and-rereview. The full history is preserved in two places:

1. The attached report file (re-attached on `--re-attach`)
2. ADO's "edited" badge on the comment shows it was modified (clicking reveals revision history if enabled at org level)

If the team prefers append-new comments, `--mode append` could be a future flag. Not in v1.

## Resurrection handling

If a slug appears in iteration N+1 that was resolved in N (i.e., it's in `prior_resolved` AND `new_slugs`):

- Treat as `remaining` (not `new`)
- Banner doesn't tag it specially
- Reasoning: the issue came back. Reviewer should investigate, but the publish skill doesn't editorialize.

If you want explicit "regression" tracking, that's a v2 feature — would need to persist a full slug-history array, not just the latest state.
